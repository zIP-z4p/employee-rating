# Обоснование выбора технологий разработки

## 1. Цель проекта

Разрабатываемый модуль предназначен для автоматизации процесса обработки, расчёта, хранения и визуализации данных по рейтингу сотрудников компании. Система должна:

- принимать рейтинговые данные по сотрудникам;
- хранить историю оценок за разные периоды;
- рассчитывать агрегированные показатели и ранги;
- показывать результаты в виде таблиц и графиков;
- предоставлять API для интеграции с внешними системами;
- поддерживать масштабирование и дальнейшее развитие.

Для реализации этих задач был выбран стек технологий, обеспечивающий:
- быстрое создание MVP;
- хорошую производительность;
- удобство сопровождения;
- масштабируемость;
- простоту локального и серверного развёртывания.

---

## 2. Выбор backend-технологий

### 2.1 Python 3.11

**Причины выбора:**
- высокий темп разработки;
- понятный и читаемый синтаксис;
- богатая экосистема библиотек для API, БД, аналитики и тестирования;
- удобство написания бизнес-логики и обработки данных;
- широкое распространение в корпоративной разработке и аналитических системах.

**Преимущества для проекта:**
- удобно реализовывать расчёт рейтингов, агрегации и валидацию данных;
- легко писать тесты;
- просто интегрировать импорт CSV, фоновую обработку и REST API.

**Почему не Java / C# / Go:**
- Java и C# дали бы более тяжёлый старт для небольшого модуля;
- Go хорошо подходит для высоконагруженных сервисов, но менее удобен для быстрой бизнес-разработки и ORM-слоя;
- Python обеспечивает лучший баланс между скоростью разработки и функциональностью.

---

### 2.2 FastAPI

**Причины выбора:**
- современный и быстрый Python-фреймворк для REST API;
- нативная поддержка асинхронности;
- автоматическая генерация Swagger/OpenAPI документации;
- тесная интеграция с Pydantic для валидации данных;
- высокая производительность по сравнению с классическими Python-фреймворками.

**Преимущества для проекта:**
- быстрое создание API для сотрудников, рейтингов, импорта и отчётности;
- встроенная документация облегчает тестирование и интеграцию;
- удобная типизация помогает уменьшить число ошибок;
- подходит для микросервисного и модульного подхода.

**Почему не Flask / Django:**
- Flask потребовал бы больше ручной инфраструктурной настройки;
- Django мощный, но для API-first сервиса избыточен;
- FastAPI лучше подходит для лёгкого, современного и типизированного API.

---

### 2.3 Pydantic

**Причины выбора:**
- строгая валидация входных и выходных данных;
- поддержка Python type hints;
- удобное описание DTO/схем запросов и ответов;
- интеграция с FastAPI “из коробки”.

**Преимущества для проекта:**
- защищает API от некорректных данных;
- упрощает описание контрактов API;
- делает структуру обмена данными прозрачной и предсказуемой.

---

## 3. Выбор технологии хранения данных

### 3.1 PostgreSQL

**Причины выбора:**
- надёжная промышленная СУБД;
- поддержка сложных SQL-запросов, агрегатов и оконных функций;
- высокая стабильность и зрелость;
- отличная поддержка транзакций и индексов;
- хорошая работа с аналитическими выборками.

**Преимущества для проекта:**
- удобно хранить сотрудников, отделы, метрики и историю рейтингов;
- можно эффективно рассчитывать ранги и агрегаты;
- оконные функции позволяют реализовать ranking и percentile на стороне БД;
- легко масштабируется вертикально и хорошо подходит для корпоративных систем.

**Почему не MySQL / MongoDB:**
- MySQL уступает PostgreSQL по удобству аналитических запросов;
- MongoDB хуже подходит для строго структурированных сущностей и связей;
- для рейтинговой системы с отношениями и историей лучше подходит реляционная модель.

---

### 3.2 SQLAlchemy 2.0

**Причины выбора:**
- популярная ORM для Python;
- поддержка как ORM-подхода, так и SQL выражений;
- поддержка async;
- хорошая совместимость с PostgreSQL.

**Преимущества для проекта:**
- позволяет хранить бизнес-сущности в структурированном виде;
- упрощает работу с БД и миграциями;
- при необходимости можно комбинировать ORM и “сырой” SQL для сложных аналитических запросов.

**Почему не Django ORM / raw SQL only:**
- Django ORM потребовал бы использования Django-стека;
- только raw SQL усложнил бы поддержку и тестирование;
- SQLAlchemy даёт гибкость и масштабируемость.

---

### 3.3 Alembic

**Причины выбора:**
- стандартный инструмент миграций для SQLAlchemy;
- поддерживает версионирование схемы БД;
- подходит для командной разработки и CI/CD.

**Преимущества для проекта:**
- изменения структуры БД становятся контролируемыми;
- можно безопасно разворачивать систему в разных окружениях;
- упрощается сопровождение проекта.

---

## 4. Выбор frontend-технологий

### 4.1 React

**Причины выбора:**
- самая распространённая библиотека для построения SPA-интерфейсов;
- компонентный подход;
- большое сообщество и экосистема;
- удобство переиспользования UI-компонентов.

**Преимущества для проекта:**
- можно быстро собрать дашборд рейтингов;
- удобно реализовать таблицы, фильтры, карточки, графики;
- легко расширять интерфейс новыми экранами.

**Почему не Angular / Vue:**
- Angular более тяжёлый для небольшого модуля;
- Vue тоже хорош, но React был выбран как более универсальный и широко используемый стек в корпоративной среде.

---

### 4.2 TypeScript

**Причины выбора:**
- статическая типизация на фронтенде;
- уменьшение числа ошибок при работе с API;
- удобство рефакторинга;
- лучшая поддержка IDE.

**Преимущества для проекта:**
- контракты API и UI лучше согласованы;
- безопаснее работа с сущностями рейтинга, сотрудника, тренда;
- проще сопровождать проект по мере роста.

---

### 4.3 Vite

**Причины выбора:**
- быстрый dev-server;
- современный инструмент сборки;
- меньшая сложность по сравнению с устаревшими инструментами;
- высокая скорость сборки фронтенда.

**Преимущества для проекта:**
- ускоряет локальную разработку;
- упрощает настройку современного frontend-окружения;
- хорошо подходит для React + TypeScript.

---

### 4.4 Axios

**Причины выбора:**
- удобный HTTP-клиент;
- понятная работа с запросами, заголовками и ошибками;
- распространённый стандарт для frontend-приложений.

**Преимущества для проекта:**
- простой вызов backend API;
- централизованная настройка клиента;
- удобно добавлять interceptors и обработку ошибок.

---

### 4.5 TanStack Query

**Причины выбора:**
- удобное управление серверным состоянием;
- кэширование запросов;
- автоматический refetch;
- упрощение логики загрузки данных.

**Преимущества для проекта:**
- удобно получать списки сотрудников, снимки рейтингов и тренды;
- сокращается boilerplate-код;
- улучшается UX за счёт кэширования и фонового обновления.

---

### 4.6 Recharts

**Причины выбора:**
- удобная библиотека графиков для React;
- простой API;
- достаточная функциональность для дашбордов;
- легко строить line/bar/pie charts.

**Преимущества для проекта:**
- быстро реализуется визуализация рейтингов;
- удобно отображать тренды сотрудников;
- снижает сложность фронтенд-разработки.

**Почему не D3 напрямую:**
- D3 мощнее, но значительно сложнее;
- для задач дашборда избыточен;
- Recharts даёт лучший баланс простоты и возможностей.

---

## 5. Выбор технологий кэширования и фоновой обработки

### 5.1 Redis

**Причины выбора:**
- быстрый in-memory storage;
- широко применяется для кэширования;
- простота интеграции;
- поддержка TTL и хранения промежуточных данных.

**Преимущества для проекта:**
- кэширование готовых снимков рейтинга;
- снижение нагрузки на PostgreSQL;
- ускорение повторных запросов к аналитике.

**Почему не Memcached:**
- Redis функциональнее;
- удобнее для расширения;
- подходит не только для кэша, но и как брокер сообщений.

---

### 5.2 Celery

**Причины выбора:**
- зрелая система фоновых задач для Python;
- интеграция с Redis;
- позволяет выносить тяжёлые операции из синхронного API.

**Преимущества для проекта:**
- можно асинхронно строить отчёты и импортировать большие файлы;
- пользователь не ждёт завершения долгой операции в HTTP-запросе;
- проще масштабировать фоновые вычисления отдельно от API.

**Почему не cron / ручная реализация:**
- Celery даёт стандартный и расширяемый подход;
- проще обрабатывать статусы задач, ретраи и очереди.

---

## 6. Выбор средств контейнеризации и развёртывания

### 6.1 Docker

**Причины выбора:**
- единое окружение для всех разработчиков;
- предсказуемость запуска;
- изоляция сервисов;
- переносимость между локальной машиной и сервером.

**Преимущества для проекта:**
- PostgreSQL, Redis, backend и frontend запускаются одинаково везде;
- упрощается установка и сопровождение;
- снижается число ошибок “у меня работает”.

---

### 6.2 Docker Compose

**Причины выбора:**
- простой способ управлять несколькими сервисами;
- удобно для локальной разработки и тестового стенда;
- позволяет быстро поднять полный стек одной командой.

**Преимущества для проекта:**
- запуск всей системы через `docker-compose up`;
- удобная настройка зависимостей между сервисами;
- простое масштабирование и добавление новых компонентов.

---

## 7. Выбор инструментов тестирования

### 7.1 Pytest

**Причины выбора:**
- стандарт де-факто для тестирования на Python;
- лаконичный синтаксис;
- поддержка фикстур;
- удобен для unit и integration тестов.

**Преимущества для проекта:**
- легко тестировать расчёт рейтингов, API и импорт данных;
- удобно организовывать тестовую инфраструктуру;
- хорошо сочетается с FastAPI и SQLAlchemy.

---

### 7.2 HTTPX

**Причины выбора:**
- современный HTTP-клиент;
- поддержка async;
- удобен для интеграционных тестов FastAPI.

**Преимущества для проекта:**
- можно проверять реальные HTTP-маршруты API;
- помогает тестировать весь слой API целиком.

---

## 8. Почему выбран именно такой стек целиком

Выбранный стек соответствует задачам проекта по следующим причинам:

1. **Быстрая разработка**  
   Python + FastAPI + React позволяют быстро собрать рабочий прототип и затем развивать его.

2. **Надёжное хранение и аналитика**  
   PostgreSQL хорошо подходит для структурированных корпоративных данных и расчёта рейтингов.

3. **Масштабируемость**  
   Backend, frontend, Redis и PostgreSQL разделены на независимые сервисы, которые можно масштабировать отдельно.

4. **Поддерживаемость**  
   Слоистая архитектура, типизация и миграции делают проект удобным для дальнейшей поддержки.

5. **Интеграция**  
   REST API и OpenAPI документация упрощают подключение внешних систем.

6. **Производительность**  
   Асинхронный backend, кэширование Redis и предрасчёт снимков рейтинга снижают нагрузку и ускоряют ответы.

---

## 9. Вывод

Для данного проекта выбран современный, практичный и хорошо сбалансированный технологический стек:

- **Python + FastAPI** — для API и бизнес-логики;
- **PostgreSQL + SQLAlchemy + Alembic** — для хранения данных и миграций;
- **Redis + Celery** — для кэширования и фоновых задач;
- **React + TypeScript + Vite** — для интерфейса визуализации;
- **Docker + Docker Compose** — для развёртывания;
- **Pytest + HTTPX** — для тестирования.

Такой выбор обеспечивает:
- высокую скорость разработки;
- надёжность;
- удобство локального запуска;
- хорошую расширяемость;
- готовность к промышленному использованию.
